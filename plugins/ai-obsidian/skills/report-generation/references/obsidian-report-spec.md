# Obsidian 报告生成规范

## 数据模型

- 默认根路径：当前项目根目录。
- 任务文件：`<项目根目录>/任务/{YYYY}/计划中.md`
- 归档文件：`<项目根目录>/任务/{YYYY}/已归档.md`
- 报告文件与任务文件放在同一年度目录。
- 任务按 `## 项目名` 分区，任务行使用 Obsidian Markdown checkbox。
- 父任务为顶层 checkbox，子任务为每层 1 个 Tab 缩进的 checkbox。

## 报告文件

| 报告 | 文件 |
| --- | --- |
| 日报 | `任务/{YYYY}/日报.md` |
| 周报 | `任务/{YYYY}/周报.md` |
| 月报 | `任务/{YYYY}/月报.md` |
| 季度报告 | `任务/{YYYY}/季度报告.md` |
| 年度报告-半年 | `任务/{YYYY}/半年报.md` |
| 年度报告-全年 | `任务/{YYYY}/年度报告.md` |

写入规则：

- 文件不存在时创建，并写入总标题：`# {YYYY} 日报`、`# {YYYY} 周报` 等。
- 文件存在时保留第一行总标题。
- 新报告插入到总标题下面，最新在最上面。
- 同一日期或同一周期已有报告时，默认仍在头部新增一条；只有用户明确要求覆盖时，才替换对应条目。

## 日期范围

| 报告 | 范围 |
| --- | --- |
| 日报 | 指定日期，默认今天 |
| 周报 | 指定日期所在工作周，周一到周五 |
| 月报 | 指定月份 |
| 季度报告 | `YYYY-Q1`、`YYYY-Q2`、`YYYY-Q3` 或 `YYYY-Q4` |
| 年度报告-半年 | `YYYY-H1` 或 `YYYY-H2` |
| 年度报告-全年 | 指定年份 |

筛选规则：

- 日期范围涉及多个年份时，读取多个年度目录。
- 日报只纳入 `🛫 <= 报告日期 <= 📅` 的任务。
- 周报、月报、季度报告和年度报告纳入范围内开始、计划完成、实际完成或持续中的任务。
- 任务已归档但 `✅` 或任务周期落在报告范围内时，必须纳入。

## 状态映射

| 标记 | 报告状态 |
| --- | --- |
| `[x]` | 已完成 |
| `[/]` | 进行中 |
| `[?]` | 阻塞 |
| `[ ]` | 未开始 |

- 完成日期来自 `✅ YYYY-MM-DD`。
- 计划完成日期来自 `📅 YYYY-MM-DD`。
- 开始日期来自 `🛫 YYYY-MM-DD`。
- 备注来自任务下方普通列表行中的 `备注：...`；兼容读取历史任务行尾的 `备注：...`。

## 日报格式

日报要精简、直观、明确。只按子任务逐条列出，不显示日期，不做表格，不写长段总结。

模板：

```markdown
## 2026-06-22 日报

【数据任务】基础依赖解析组件开发：当前使用 a 技术解决，目前进行中
【系统管理】报告技能重构：已确认日报与周报格式
```

生成规则：

- 项目按项目名排序。
- 每条优先合并同项目、同父任务或同类工作的相关子任务；如果父任务没有子任务，则把父任务当作一条任务。
- 格式固定为：`【项目名】子任务标题：描述`
- 描述优先来自备注；没有备注时，用状态和任务上下文生成一句简短说明。
- 描述必须是一句短句，避免解释背景、日期和统计；单条描述优先控制在 20-35 个汉字左右，日报一般控制在 3-5 条。
- 不显示 `🛫`、`📅`、`✅` 日期。
- 不输出未落在当天任务周期内的任务。

## 周报格式

周报要具体到外部读者能理解任务情况。正式格式使用 AnyBlock `[list2table]` 三列表格：`项目 | 任务 | 完成情况`。项目作为父节点，项目下的顶级任务作为子节点，项目列由 AnyBlock 父子层级自动跨行合并；任务列写具体任务事实，完成情况列写父任务本周总体概括。

模板：

```markdown
## 2026-06-22 ~ 2026-06-26 周报

[list2table]
- < 项目| 任务| 完成情况
- 项目A
  - **【进行中】 顶级任务一**<ul><li>任务组【完成日期：YYYY-MM-DD】。<ul><li>已完成子任务说明。</li><li>已完成子任务说明。</li></ul></li><li>子任务组【进行中：30%】<ul><li>当前进展说明。</li></ul></li></ul> | <ul><li>本周完成：概括完成内容或效果。</li><li>本周进行中：概括当前阶段或情况。</li><li>后续安排：概括下周推进方向。</li></ul>
  - **【已完成】 顶级任务二**<ul><li>任务说明【完成日期：YYYY-MM-DD】。</li></ul> | <ul><li>本周完成：概括完成内容或效果。</li><li>本周进行中：无。</li><li>后续安排：按任务记录整理后续安排。</li></ul>
```

生成规则：

- 标题使用工作周日期范围：`## YYYY-MM-DD ~ YYYY-MM-DD 周报`，通常为周一到周五。
- 标题下面不要再单独写“时间范围”或重复日期范围行。
- 日报和周报默认只读取 `计划中.md`；不扫描 `已归档.md`，避免部分归档父任务重复统计。
- 不统计 `[ ]` 未开始任务。
- 数据扫描范围必须同时考虑完成时间和规划时间：任务或子任务只要 `✅` 落在本周，或计划区间与本周有交集（`🛫 <= 周五` 且 `📅 >= 周一`），就纳入候选。
- 周报行粒度必须是项目下的顶级任务；不要把子任务拆成独立表格行。
- 顶级任务命中条件：顶级任务本身命中扫描范围，或任一子任务命中扫描范围。
- AnyBlock 表头必须写成单行：`- < 项目| 任务| 完成情况`。
- 项目列合并必须用 AnyBlock 父子层级实现：父节点是项目，子节点是顶级任务行。
- 子节点使用 `任务内容 | 完成情况内容` 填充后两列。
- 任务标题格式：`**【状态】 顶级任务标题**`。状态使用中文符号 `【】`，例如 `【进行中】`、`【已完成】`、`【阻塞】`。
- 任务列负责具体事实：写具体做了哪些任务、当前哪些在进行、任务进度、完成日期、计划观察或计划完成信息。
- 任务列必须保留已有的任务状态、进度和完成时间：完成日期写 `【完成日期：YYYY-MM-DD】`，进行中进度写 `【进行中：NN%】`，没有百分比但有计划日期时写 `【进行中：计划YYYY-MM-DD】` 或按任务记录写当前阶段。
- 进度百分比只能来自任务记录或用户已明确给出的报表上下文；不要自行推算或编造百分比。
- 任务列和完成情况列内部暂用 HTML 无序列表 `<ul><li>...</li></ul>` 保证 AnyBlock 表格渲染稳定；子任务多时必须拆成多条列表，不要堆在一行。
- 多层任务使用嵌套 `<ul>`：顶级任务标题在表格行开头，一级任务组是第一层 `<li>`，具体子任务或说明是第二层 `<li>`。
- 子任务行一般不重复显示 checkbox 状态；只有完成日期、进度或当前阶段对周报有价值时，才在对应条目后用 `【】` 补充。
- 完成情况列负责父任务本周总体概括，默认用三条：`本周完成：...`、`本周进行中：...`、`后续安排：...`。
- 完成情况列不要写太细，不逐条复制任务列；只总结本周完成、进行中和后续安排。
- 任务明细和完成情况只能来自任务标题、任务下方说明、相关子任务标题/说明、状态和日期关系；不要编写原任务中没有的泛化事实。
- 周报允许适当整理措辞，但只能基于任务标题、父任务、项目、说明、状态和日期关系进行解释。
- 不编造完成结果、完成比例、业务指标或不存在的依赖方。
- 同项目内按父任务层级、子任务日期和状态排序：已完成、进行中、阻塞。

### 周报测试规则

- 测试阶段先使用增加模式生成候选周报，不覆盖已有正式周报段落。
- 候选版经用户确认后，才能替换正式周报或删除旧候选版本。

## 月报格式

模板：

```markdown
## 2026-06 月报

### 项目A

- 主要完成：
  - 任务一：基于任务记录归纳成果
- 持续推进：
  - 任务二：说明当前进展
- 问题与风险：
  - 任务三：说明阻塞或待确认事项
- 下月关注：
  - 从未完成、进行中和后续计划任务中归纳
```

## 季度报告格式

季度报告只用于 `Q1/Q2/Q3/Q4` 四类季度周期，默认写入 `任务/{YYYY}/季度报告.md`。正式输出采用材料化结构，不使用表格，按项目分段。

模板：

```markdown
## 2026-Q2 季度报告

### 项目A

- 围绕业务贡献总结季度完成内容，并把可衡量指标融入描述中，例如完成任务数、验证规模、接口请求量、问题修复数量。
- 说明当前仍在推进或需要下季度继续关注的事项。

### 下季度规划

- 按主要方向写规划，说明目标、重点工作、预期贡献和建议跟踪指标。
```

生成规则：

- 标题使用 `## YYYY-QN 季度报告`。
- 周期只能是 `Q1`、`Q2`、`Q3`、`Q4`；不要把半年、全年或任意日期范围写成季度报告。
- 必须按项目归纳，不逐条罗列全部任务。
- 重点突出业务贡献和可参考指标；指标必须来自任务记录或用户明确提供的信息。
- 可参考指标要融合到描述中，不单独列“指标清单”。
- 如果用户要求规划，按方向组织，例如“DSP框架统一迁移”“AI框架完善升级”，写目标、重点工作、预期贡献和建议跟踪项。
- 不编造完成比例、业务收益、修复数量或接口量；任务中没有明确指标时，只做定性总结。

## 年度报告格式

年度报告分为半年报告和全年报告：半年报告写入 `任务/{YYYY}/半年报.md`，全年报告写入 `任务/{YYYY}/年度报告.md`。两者都按项目归纳，采用材料化要点风格，不写流水账。

### 半年报告格式

模板：

```markdown
## 2026-H1 半年报告

### 一、项目A

- **核心方向：**
  - 用截图式材料风格总结重点工作，融合完成任务数、验证规模、接口量、问题修复数量等可参考指标。
- **业务贡献：**
  - 说明对业务稳定、迁移效率、数据质量或平台能力的支撑。
- **不足与后续：**
  - 说明未闭环事项和下半年改进方向。

### 亮点、不足以及改进计划

- **亮点：** ...
- **不足：** ...
- **改进计划：** ...
```

生成规则：

- 标题使用 `## YYYY-H1 半年报告` 或 `## YYYY-H2 半年报告`。
- 默认按项目分段，使用“编号标题 + 粗体小项 + 嵌套要点”的材料风格。
- 每个项目段落要精炼；用户要求字数时严格按用户给出的上下限处理。
- 可参考指标必须融入对应项目描述中，不单独罗列指标清单。
- 半年自评可额外生成“亮点、不足以及改进计划”三项模式；用户要求最低字数时，三项合计必须达到最低字数。
- 不编造任务事实或量化指标；只使用任务记录和用户明确补充的信息。

### 全年报告格式

模板：

```markdown
## 2026 全年报告

### 项目A

- 年度成果：
  - 基于已完成任务归纳
- 建设与支撑：
  - 基于父子任务层级归纳
- 问题与风险：
  - 基于阻塞和待确认任务归纳

### 下一年度计划

- 从未完成任务和后续计划中整理
```

生成规则：

- 标题使用 `## YYYY 全年报告`。
- 按项目归纳全年成果、建设支撑、问题风险和下一年度计划。
- 延续半年报告的材料化风格：分项目、要点化、指标融入正文。
- 全年报告不可写成季度报告或半年报告；如用户只给 `H1/H2`，应生成半年报告。

## 质量要求

- 日报：短、直观、明确。
- 周报：具体、可读、让不了解上下文的人能理解。
- 月报及以上：按项目归纳，不逐条堆砌全部任务。
- 不编造任务、完成日期、结果、指标或进度百分比。
- 如果任务记录没有量化指标，不写量化指标。
- 如果任务描述明显缺失，允许用父任务、项目和状态补足上下文，但不能改变事实。
